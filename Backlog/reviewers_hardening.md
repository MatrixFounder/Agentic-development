**TASK: Доказуемость «чистого» прохода Code Reviewer и объективный критерий выхода Сарказмотрона (кросс-вендорно)**

**Что именно дорабатываем.**
1. **Code Reviewer:** при отсутствии критичных замечаний отчёт обязан нести след проверки (что проверено) — чтобы «не смотрел» отличалось от «посмотрел и чисто». Сейчас следа нет.
2. **Code Reviewer (контракт):** привести поля вывода SOT в соответствие с тем, что реально потребляет оркестратор, устранив дрейф из четырёх источников.
3. **Сарказмотрон:** критерий одобрения переписать с «вынужден выдумывать придирки» на объективное условие выхода.

Роли **не сливаем и не выравниваем** — разные цели, обе работают как надо. Маршрутизацию и фазы (`/vdd-start-feature`, `/vdd-plan`, `/vdd-develop`, диспетч, петли, лимит 3-REJECT, HITL) **не трогаем**.

**Контракт с оркестратором (соблюсти строго).**
- Несущее поле control-flow — `has_critical_issues`. Его имя, тип и семантику **не менять ни при каких правках**. DECISION TABLE в `skill-orchestrator-patterns` не редактируем.
- Прозовый отчёт оркестратор передаёт разработчику для фиксов; блок «Verified» живёт **в теле отчёта** и на control-flow не влияет.
- Любые добавления в структурированный вывод — **только аддитивные**; нельзя ломать ключи, которые ждут orchestrator-схема и обёртка (`has_critical_issues`, `e2e_tests_pass`, `stubs_replaced`).

**Доработка 1 — след проверки у Code Reviewer.**

*Сейчас.* Шаг «Artifact Creation» в `09_code_reviewer_prompt.md` требует Header / Assessment / Comments / Final Decision. Когда критичных замечаний нет, на выходе нет ничего, что показывало бы объём проверки: пустой и подтверждённый проход неразличимы.

*Должно стать.* В формат отчёта добавить **обязательный блок «Verified»** (plain markdown в теле, не в структурированном выводе): какие требования TASK сверены и какие edge-cases рассмотрены. Появляется при `has_critical_issues = false`. Тиры (🔴/🟡/🟢), комплаенс-фокус и роль гейта качества **сохраняются** — злее не делаем.

**Доработка 2 — согласование контракта вывода ревьювера.**

*Сейчас.* Контракт вывода ревьювера определён в **четырёх** местах, и они рассогласованы:
- **A — SOT** `System/Agents/09_code_reviewer_prompt.md` эмитит `{review_status, has_critical_issues}` (2 поля).
- **B — orchestrator-схема** `.agent/skills/skill-orchestrator-patterns/SKILL.md` (Extended Schema — Code Reviewer) ждёт `{comments, has_critical_issues, e2e_tests_pass, stubs_replaced}`.
- **C — обёртка** `.claude/agents/code-reviewer.md` ждёт `{review_status, has_critical_issues, stubs_replaced, e2e_tests_pass}`.
- **D — оркестратор** `System/Agents/01_orchestrator.md` (шаг 11 «Code Review») ждёт `Expected: { comments, has_critical_issues, e2e_tests_pass, stubs_replaced }`.

*Должно стать.* Привести все четыре определения к одному совместимому **структурированному суперсету**: `has_critical_issues`, `e2e_tests_pass`, `stubs_replaced`, `review_status` (`review_status` оставить — безвреден, используется обёрткой/людьми). SOT и обёртка эмитят этот суперсет; B и D приводятся к нему же. Поле `comments` — это **прозовое тело отчёта** (передаётся разработчику для фиксов), а **не структурированный ключ**: в B и D переформулировать `comments` как «прозовый отчёт», убрав его из перечня JSON-ключей. Добавления только аддитивны; семантику `has_critical_issues` не менять.

**Доработка 3 — критерий выхода Сарказмотрона (оверлей + все авторитетные определения критерия выхода).**

*Сейчас.* Выход («Hallucination Convergence») триггерится по признаку «аудитор вынужден изобретать незначащие придирки → одобрить». Это делает фабрикацию придирок *триггером* одобрения; ленивая/сикофантная модель выходит рано, признак ненаблюдаем. Критерий **определён авторитетно в четырёх местах** (а не только в оверлее): `vdd-03-develop.md`, `.agent/skills/vdd-adversarial/SKILL.md`, `.agent/skills/vdd-sarcastic/SKILL.md`, `.agent/skills/vdd-adversarial/references/vdd-methodology.md`. Правка только оверлея оставит скиллы (более авторитетный кросс-вендорный SOT) проповедовать старое правило — это воссоздаёт ровно тот многоисточниковый дрейф, который убирает Доработка 2.

*Должно стать.* Объективное условие применяется к **каждому из четырёх определений** (скиллы — через `skill-enhancer`): одобрять только когда (1) полный прогон тестов выполнен, (2) ноль CRITICAL, (3) ноль законных находок по логике/безопасности/slop, (4) осталось лишь bikeshedding/стиль. Враждебный тон и стойка «считай сломанным, пока не доказано» **сохраняются по духу**. Механика VDD-петли (3-REJECT, эскалация, HITL) не трогается. Дополнительно — освежить устаревший **ярлык** «Hallucination Convergence» в ссылающихся файлах (`vdd-05-run-full-task.md` — дайджест HITL; `System/Docs/WORKFLOWS.md` — описание правила) под новый критерий; **только терминология, без правки механики петли**.
